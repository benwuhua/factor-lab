# Factor Lab V2 Execution Blueprint

> Current execution blueprint for turning `factor-lab` into a standalone alpha-to-paper engine.

## 1. Goal

`factor-lab` should become a daily, repeatable system that can discover factors, approve them with evidence, turn them into explainable signals, build target portfolios, run risk checks, simulate paper trading, and feed the results back into factor governance.

This blueprint replaces the earlier "build the missing stages" framing. The repo now already contains the first versions of alpha governance, signal generation, portfolio construction, risk checks, execution calendar, and paper workflow. The next job is to make those pieces run together reliably.

The target loop is:

```text
AI factor research
  -> factor review and approval
  -> daily signal build
  -> target portfolio
  -> risk gate
  -> paper orders and fills
  -> reconciliation
  -> monitoring and factor promotion/demotion
```

## 3. Product Boundary

`factor-lab` should remain a standalone research-to-paper engine. It should own:

- data and provider sanity checks
- factor search
- factor approval and demotion
- signal generation
- portfolio construction
- risk checks
- paper order and fill simulation
- historical paper batches
- monitoring reports
- run manifests and audit bundles

It should not become a dashboard product, broker GUI, or discretionary stock-picking notebook. Operator experience can remain file-first and script-first until the paper workflow proves useful.

## 4. Daily Operating Chain

The daily chain should be a single reproducible path:

```text
select_factors
  -> build_daily_signal
  -> build_target_portfolio
  -> generate or simulate paper orders
  -> reconcile paper state
  -> write run bundle
```

### Standard artifacts

Each run should eventually produce:

```text
runs/YYYYMMDD/
  manifest.json
  approved_factors.yaml
  signals.csv
  signal_summary.md
  target_portfolio.csv
  target_portfolio_summary.md

  orders.csv

```

Until the run bundle is fully wired, equivalent files under `reports/` are acceptable. The important rule is that every daily run must be reproducible from its manifest and configs.

## 5. Factor Governance

The approved factor pool is the boundary between research and production-like workflows.

### Factor states

- `candidate`: generated by AI or manual research, not trusted
- `review`: passed the first oracle, needs stability checks
- `approved`: has evidence across horizon, years, regime, and universe
- `active`: allowed to influence the daily signal
- `shadow`: calculated and monitored, but not allowed to influence main score
- `retired`: kept for history, no longer used

### Promotion rules

A factor can move from `review` to `approved` only when it has:

- neutralized H20 evidence
- yearly split diagnostics
- regime split diagnostics
- universe comparison if it claims broad use
- turnover sanity
- redundancy assignment against approved factors
- a written review note

### Active rules

Approved does not mean active. Initial active factors should be conservative:

- active core:
  - `high_mean60_discount_volume_divergence_reversal_20_60_v1`
  - `intraday_volatility_skew_20_v1`
- active challenger with reduced weight:
  - `quiet_close_range_divergence_20_v1`
- shadow or reserve:
  - all remaining approved factors until daily/paper evidence supports promotion

## 6. Daily Signal Design

The signal layer should be transparent before it is clever.

### Required behavior

- load `reports/approved_factors.yaml`
- compute factor exposures for the run date
- normalize factor values cross-sectionally
- apply approval-status weights
- apply regime gates
- reduce same-family duplication
- emit per-stock factor contributions
- keep model score optional and non-dominant

### Initial weighting

Recommended first production-like configuration:

| status | base weight |
|---|---:|
| core | 1.0 |
| challenger | 0.35 to 0.5 |
| reserve | 0.0 for main score, still shadowed |
| regime_only | only active when regime matches |

Family-level score should be capped so that F001 variants do not overwhelm distinct families.

### Signal outputs

Minimum signal columns:

- `date`
- `instrument`
- `tradable`
- `rule_score`
- `model_score`
- `ensemble_score`
- `active_regime`
- `top_factor_1`
- `top_factor_1_contribution`
- `top_factor_2`
- `top_factor_2_contribution`
- `risk_flags`

## 7. Portfolio And Risk

The first portfolio should stay boring on purpose. The goal is not optimization; the goal is to expose whether the signal survives basic trading constraints.

### Initial portfolio rules

- rank by `ensemble_score`
- TopK: 20
- cash buffer: 5%
- max single name weight: 8%
- max new buys per day: 5
- dropout rank: 30
- equal weight after constraints

### Risk checks

The risk gate should fail closed. Minimum checks:

- min positions
- max single name weight
- max turnover
- min signal coverage
- tradability coverage
- missing price or volume fields
- zero volume names
- blacklist or manual lock list when available

If risk fails, the system should still write reports, but should not proceed to paper orders for that date.

## 8. Paper Workflow

Paper trading is the first real test of whether the research is operationally useful.

### Paper run responsibilities

- compare target portfolio with current paper holdings
- generate orders
- apply lot size and min order value
- simulate conservative fills
- include slippage and costs
- update paper positions
- reconcile expected and simulated state
- write a daily run bundle

### 30-day acceptance criteria

Before any live-like use, require at least 30 trading days of paper evidence:

- no silent data failures
- no unexplained reconciliation mismatch
- risk failures are explicit and understandable
- turnover stays within configured bounds
- top holdings and factor contributions are explainable
- paper PnL and drawdown are reported daily
- active factors have contribution tracking

The first 30 days are primarily a reliability test, not a performance victory lap.

## 9. AI Research System

Autoresearch should become three separate lanes. Keep their ledgers and promotion rules separate.

### Lane 1: Expression factors

Purpose:

- search continuous cross-sectional formulas
- optimize for neutralized IC, stability, and turnover sanity

Current status:

- already working through `run_expression_codex_loop.py`
- can run overnight with Codex CLI
- writes expression ledger and run artifacts

Next upgrades:

- stronger duplicate detection
- family-aware exploration quotas
- nightly shortlist generation
- automatic "promote candidate / keep watching / discard" summaries

### Lane 2: Pattern factors

Purpose:

- research event-like or setup-like factors such as `wangji-factor1`
- evaluate trigger precision rather than pure IC

Primary metrics:

- event count
- forward return
- win rate
- payoff
- MFE
- MAE
- trigger clustering
- whether high scores line up with human-labeled chart windows

Expected ledger:

```text
reports/autoresearch/pattern_results.tsv
```

### Lane 3: Combo factors

Purpose:

- combine already approved factors
- tune family weights and regime gates
- optimize signal and paper behavior, not raw expression novelty

Primary metrics:

- signal stability
- portfolio turnover
- paper return and drawdown
- contribution concentration
- factor family diversity
- robustness across regimes

Combo research should never rewrite the evaluation contract or create unreviewed raw formulas. It works only above the approved factor layer.

## 10. Monitoring And Feedback

Every daily run should produce enough evidence to answer:

- which factors drove today's top names
- whether active factors are still behaving like their research evidence
- whether paper performance is coming from one family or diversified signal
- whether risk failures are recurring
- whether challengers deserve promotion
- whether active factors should be demoted

### Weekly review

Once per week:

- summarize active factor contributions
- compare active vs shadow factors
- inspect top names and turnover
- review risk failures
- decide promote, demote, or keep watching

### Monthly review

Once per month:

- refresh approved factor governance
- rerun stability checks for active factors
- retire redundant variants
- update signal weights only with written evidence

## 13. What Not To Do Yet

- Do not add broker automation.
- Do not let the model score dominate before rule scores have a stable paper baseline.
- Do not keep promoting variants from the same family without new economic meaning.
- Do not optimize weights against one short paper window.
- Do not treat one good IC table as deployable evidence.
- Do not add dashboard work before the file-first daily workflow is reliable.

## 14. Ready Criteria

`factor-lab` is ready to influence real allocation decisions only when:

- daily run bundles are reproducible
- approved factors have current governance evidence
- active factors are explainable at stock level
- risk gates fail closed
- paper workflow has 30 stable trading days
- reconciliation is reliable
- paper behavior matches or explains deviations from research assumptions
- promotion and demotion decisions are documented

Until then, the system is a research and paper engine, not a live trading engine.
